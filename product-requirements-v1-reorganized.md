# AgentLens 需求重整与功能拆分文档

版本：v1.0-reorganized  
日期：2026-05-06  
来源：`AgentLens产品需求文档PRDv1.0_整理版.docx` 与当前代码实现对照

---

## 1. 文档目标

本文档将原 PRD 中较大的产品愿景，结合当前仓库已经具备的能力，重新整理为一组可以独立开发、独立闭环、独立验证的需求和功能包。

每个功能包必须满足三个条件：

- **可闭环**：用户或运营方可以完成一个完整动作，而不是只看到一个孤立组件。
- **可独立**：功能边界清晰，尽量不依赖多个未完成模块同时上线。
- **可验证**：有明确验收标准，可以通过页面操作、合约调用、接口测试或数据检查确认是否完成。

---

## 2. 基于当前实际情况的产品判断

### 2.1 当前代码已经较强的部分

当前项目已经具备较完整的“可信 Agent 审计市场”底座：

- 链上 Agent 注册、质押、审计请求、审计结果、申诉与信誉记录。
- 沙箱审计、报告生成、证据链、TEE attestation 与链上写回流水线。
- 市场合约支持租赁、买断和访问权判断。
- 评论合约支持基于访问权的六维评分与评论哈希。
- 前端已具备首页列表、基础筛选、Agent 详情、审计历史、审计报告等浏览路径。

### 2.2 当前缺口最大的部分

原 PRD 的新定位是“面向终端用户的 Agent 发现、比较、验证与起步平台”。当前代码在这一层仍较弱：

- 缺少面向任务的推荐流程。
- 缺少 Agent 横向 Compare。
- 缺少重点维护 Agent 与普通收录 Agent 的内容数据模型。
- 缺少安装/接入导航。
- 缺少持续更新的外部视图。
- 平台原生 Agent 的发布、审计、租赁、评论虽然有底层能力，但前端产品闭环不完整。

### 2.3 重整后的产品定位

AgentLens 不应直接从“链上审计市场”跳到“大而全 Agent 平台”，而应采用双层结构：

1. **前台：Agent 选型决策平台**
   - 帮助用户搜索、筛选、比较、理解风险，并知道如何开始使用。

2. **后台：平台原生 Agent 信任与交易能力**
   - 复用现有注册、沙箱审计、报告、信誉、评论、租赁能力，作为高等级信任与平台内交易闭环。

这样可以保留当前代码优势，同时把 PRD 的用户价值落到更清晰的功能边界中。

## 3. 需求拆分原则

### 3.1 功能包拆分标准

每个功能包都按以下格式描述：

- **用户问题**：解决谁的什么问题。
- **功能范围**：这一期做什么，不做什么。
- **当前基础**：现有代码能复用什么。
- **交付物**：需要新增或改造的页面、组件、数据、合约或服务。
- **验收标准**：如何确认完成。
- **依赖关系**：依赖哪些前置能力。
- **优先级**：P0、P1、P2。

### 3.2 优先级定义

- **P0：MVP 必须完成**。没有它，产品无法体现“选型决策平台”的核心价值。
- **P1：闭环增强**。让平台从“可看”变成“可用、可信、可互动”。
- **P2：增长与规模化**。提升留存、运营效率和商业化能力。

---

## 4. MVP 用户主路径

重整后的 MVP 应优先保证以下路径可用：

```text
用户进入首页
  -> 搜索或选择场景
  -> 获得候选 Agent 列表
  -> 查看关键风险和信任层级
  -> 加入 Compare 或进入详情页
  -> 查看适用场景、风险说明、安装导航、证据记录
  -> 跳转官方开始使用，或查看平台原生 Agent 的租赁/试用入口说明
```

这条路径中的每一步都必须能单独验收。

说明：MVP 阶段必须让用户看懂平台原生 Agent 的租赁/试用能力，但不强制完成真实链上租赁交易。真实租赁、购买和访问权交易闭环由 F-012 在 P1 阶段完成。

## 6. P0 功能包

### F-001 Agent 内容数据模型

**用户问题**

当前前端主要依赖链上 registry 扫描数据，适合展示平台原生 Agent，但无法承载 PRD 要求的重点维护 Agent、普通收录 Agent、安装导航、外部观察等内容。

**功能范围**

建立统一 Agent 内容模型，至少支持三类 Agent：

- `curated`：重点维护 Agent。
- `listed`：普通收录 Agent。
- `native`：平台原生 Agent，来自链上注册和审计能力。

第一版可以使用静态 JSON 或 TypeScript 数据文件，不强制建设后台 CMS。

**当前基础**

- 前端已有 `useAgentList` 与 Agent 卡片结构。
- 链上已有平台原生 Agent profile、audit、reputation、pricing。

**交付物**

- `AgentCatalogEntry` 数据结构。
- 重点维护 Agent 样例数据，建议先维护 10-20 个，而不是一次性追求 50-80 个。
- 链上 native Agent 与内容模型的合并逻辑。
- 基础字段：名称、简介、分类、标签、适用场景、不适用场景、风险等级、接入方式、安装导航、官方链接、维护类型、信任层级。

**验收标准**

- 首页和列表页能同时展示 curated、listed、native 三类 Agent。
- 每个 Agent 卡片能展示其类型、核心场景、风险等级、信任层级。
- native Agent 仍能展示链上审计和价格信息。
- curated/listed Agent 没有链上记录时，不出现空白或错误状态。

**依赖关系**

无。该功能是后续推荐、Compare、详情页重构的基础。

---

### F-002 首页定位与入口重构

**用户问题**

用户首次进入平台时，需要快速理解 AgentLens 是帮助自己“选 Agent 并开始使用”的工具，而不是只看懂底层链上审计技术。

**功能范围**

首页重构为决策入口：

- 主标题强调“找到适合你的 Agent，并知道怎么开始”。
- 搜索框支持名称、场景、任务关键词。
- 展示热门场景入口。
- 展示重点维护 Agent。
- 展示平台原生 Agent 入口。
- 保留信任能力说明，但降级为辅助模块。

**当前基础**

- `HomePage` 已有首页结构。
- `MarketplaceHero` 已有统计条。
- `LeaderboardSection` 已有排行榜。
- `SearchFilterBar` 已有搜索框和筛选。

**交付物**

- 新首页 Hero 文案。
- 场景入口组件。
- 重点维护 Agent 区块。
- 平台原生 Agent 区块。
- “为什么比直接问大模型更适合做选型”说明模块。

**验收标准**

- 首屏不再以链上、TEE、审计作为唯一主叙事。
- 用户可从首页直接进入搜索、场景列表或平台原生 Agent 列表。
- 无链上数据时，首页仍能展示 curated/listed 内容。
- 通过人工走查，用户在 10 秒内能回答：这个平台做什么、我下一步该点哪里。

**依赖关系**

依赖 F-001。

---

### F-003 搜索与筛选体验

**用户问题**

用户面对多个 Agent 时，需要在 1-2 分钟内筛出一批候选，而不是浏览长列表。

**功能范围**

在现有搜索筛选基础上补齐 PRD 关键维度：

- 场景。
- Agent 类型。
- 接入方式。
- 是否重点维护。
- 是否平台原生。
- 是否支持平台租赁。
- 是否有审计报告。
- 风险等级。
- 使用复杂度。
- 是否有安装导航。

第一版可继续放在首页列表区，不强制独立 `/search` 路由。

**当前基础**

- `SearchFilterBar` 已支持状态、风险、任务类型、价格、TEE、排序。
- `HomePage` 已有 filter/sort 逻辑。

**交付物**

- 扩展筛选字段。
- Agent 卡片展示更多决策摘要。
- “清空筛选”和结果数量提示。
- 结果为空时给出替代建议。

**验收标准**

- 搜索关键词可以匹配名称、简介、标签、场景。
- 用户可筛出“平台原生 + 可租赁 + 有审计报告”的 Agent。
- 用户可筛出“低复杂度 + 有安装导航”的 Agent。
- 筛选结果卡片展示关键信息，而不是只有名称和分数。

**依赖关系**

依赖 F-001。

---

### F-004 Agent 决策型详情页

**用户问题**

用户进入 Agent 详情页后，需要知道“适不适合我、风险是什么、怎么开始”，而不是只看到链上元数据。

**功能范围**

详情页拆成以下模块：

- 基本信息。
- 适用场景。
- 不适用场景。
- 推荐用户。
- 风险说明。
- 安装/接入导航。
- 信任与证据。
- 平台原生能力。

**当前基础**

- `AgentDetailPage` 已展示 profile、pricing、reputation、risk、trust guarantee、latest audit、audit history、reviews。
- `RiskProfileCard` 已能基于维度分推导场景适配。

**交付物**

- 决策摘要卡。
- 适用/不适用场景模块。
- 风险说明模块。
- 推荐用户模块。
- 对 native Agent 展示审计、租赁、评论、信誉、历史记录。
- 对 curated/listed Agent 展示官方链接和平台维护内容。

**验收标准**

- 详情页顶部能明确给出“适合谁 / 不适合谁 / 建议下一步”。
- 用户不阅读审计报告也能理解主要风险。
- curated/listed/native 三类 Agent 都能打开详情页。
- native Agent 的链上审计信息仍可继续查看。

**依赖关系**

依赖 F-001、F-006。

---

### F-005 安装与接入导航

**用户问题**

用户知道某个 Agent 可能适合自己后，仍然不知道从哪里开始、是否需要本地安装、第一步应该做什么。

**功能范围**

为每个重点维护 Agent 提供平台整理的起步导航：

- 使用方式：本地、云端、API、浏览器插件、CLI、SaaS。
- 前置条件。
- 第一步操作。
- 基础接入步骤。
- 官方文档链接。
- 平台建议起步路径。
- 常见坑或风险提醒。

第一版不要求覆盖所有 Agent，优先覆盖 curated Agent。

**当前基础**

当前没有面向终端用户的安装导航。`ConfigPanel` 仅是开发环境配置说明，不能替代该功能。

**交付物**

- 安装导航数据结构。
- 详情页安装导航模块。
- 列表卡片上的“有安装导航”标记。

**验收标准**

- 至少 10 个重点维护 Agent 有完整安装导航。
- 用户可以在详情页看到第一步操作和官方入口。
- 安装导航不是单纯外链，必须包含平台自己的起步建议。
- 搜索筛选可筛出“有安装导航”的 Agent。

**依赖关系**

依赖 F-001、F-004。

---

### F-006 信任层级映射

**用户问题**

用户需要快速理解某个 Agent 的可信度依据，而不是被审计状态、TEE、信誉分、评论等多个概念分散注意力。

**功能范围**

统一定义四个信任层级：

- **Tier 0：已收录**。有基础信息。
- **Tier 1：已观察**。平台维护了结构化外部视图。
- **Tier 2：已验证**。有平台审计、发布者声明或可验证证据。
- **Tier 3：高等级验证**。有完整信任报告、attestation、链上记录等。

第一版可以通过规则映射计算，不要求新增合约字段。

过渡期规则：在 F-009 尚未完成前，Tier 1 不依赖完整时间线，只依赖 `AgentCatalogEntry` 中的 `latestObservedAt` 和 `observationSummary` 两个基础字段。F-009 完成后，再把 Tier 1 的证据从单条观察摘要扩展为完整外部视图更新记录。

**当前基础**

- 链上已有 audit status、attestation hash、report hash、evidence root、reputation。
- 前端已有 TEE verified、audit status、risk、reputation 展示。

**交付物**

- Trust Tier 计算规则。
- Agent 卡片上的 Tier badge。
- 详情页上的 Tier 解释。
- 审计报告与 Tier 之间的关联说明。

**验收标准**

- 每个 Agent 都有一个 Tier 状态。
- 用户能看到该 Tier 的原因。
- Tier 1 必须至少有最近观察时间和观察摘要。
- native Agent 通过审计并有 attestation/report 时，可进入 Tier 3。
- curated/listed Agent 没有链上记录时，不会被误标为高等级验证。

**依赖关系**

依赖 F-001。

---

### F-007 Agent 横向 Compare

**用户问题**

用户在多个候选 Agent 之间切换详情页会增加决策成本，需要一个直观比较页面。

**功能范围**

支持 2-4 个 Agent 横向对比：

- 核心功能。
- 适用场景。
- 不适用场景。
- 安装/接入难度。
- 风险等级。
- 平台维护深度。
- 是否有审计/验证。
- 是否支持租赁/试用。
- 是否平台原生。
- 起步建议。

第一版 Compare 使用本地状态或 URL 参数即可。

**当前基础**

当前未发现 Compare 路由、组件或状态。

**交付物**

- Agent 卡片上的“加入对比”按钮。
- Compare 托盘或候选栏。
- Compare 页面。
- 差异高亮和推荐结论。

推荐结论的第一版规则：

- 安装复杂度最低且风险不高的 Agent 标记为“建议先试”。
- 信任层级最高、审计或观察证据最完整的 Agent 标记为“适合正式接入”。
- 风险高、缺少安装导航或缺少可信证据的 Agent 标记为“暂不建议当前使用”。
- 如果规则无法给出唯一结论，页面应说明“需要人工判断”，而不是强行推荐。

**验收标准**

- 用户可从列表加入 2-4 个 Agent。
- Compare 页面可通过 URL 复现。
- 对比结果突出差异，不只是字段平铺。
- 页面给出“更适合先试 / 更适合正式接入 / 不建议当前使用”的结论。

**依赖关系**

依赖 F-001、F-004、F-005、F-006。

## 7. P1 功能包

### F-008 需求输入到候选推荐

**用户问题**

用户不一定知道该搜索哪个 Agent 名称，更常见的是只知道自己要完成什么任务。

**功能范围**

提供一个轻量需求表单：

- 想做什么任务。
- 希望连接什么系统。
- 个人还是团队使用。
- 更看重安全、易用、能力还是价格。
- 是否愿意尝试平台原生 Agent。
- 是否需要低门槛开始。

第一版使用规则引擎和结构化字段匹配，不要求接入复杂 AI 推荐。

第一版规则边界：

- 只基于平台已有结构化字段推荐，不生成未维护过的事实。
- 若候选不足 3 个，应显示“候选不足”并解释缺少哪些条件。
- 需要维护一组固定 golden cases，用于验证同一输入能稳定返回预期候选。

**当前基础**

当前没有推荐页或推荐状态。

**交付物**

- 需求输入页面或弹窗。
- 推荐规则。
- 推荐结果页。
- 每个推荐项的推荐理由、风险提示、建议下一步。
- 加入 Compare 按钮。

**验收标准**

- 用户提交需求后，返回 3-5 个候选 Agent。
- 每个候选都有推荐理由和风险提示。
- 推荐结果明确标注“建议先试哪个”。
- 推荐结果可以加入 Compare。
- 至少 5 条 golden cases 的推荐结果稳定可复现。

**依赖关系**

依赖 F-001、F-003、F-007。

---

### F-009 持续更新外部视图

**用户问题**

用户需要知道平台是否持续维护信息，而不是看到一次性的静态介绍。

**功能范围**

为重点维护 Agent 提供外部观察记录：

- 最近观察时间。
- 最近能力变化。
- 最近文档变化。
- 最近价格变化。
- 最近风险提示变化。
- 平台推荐策略变化。

第一版支持人工维护即可。

**当前基础**

当前没有外部视图更新记录模型。

**交付物**

- 外部观察记录数据结构。
- 详情页“最近观察”模块。
- Agent 卡片上的更新时间标记。
- 搜索排序支持“最近观察”。

**验收标准**

- 重点维护 Agent 展示最近观察时间。
- 用户可看到最近一次变化内容。
- 没有观察记录的普通收录 Agent 显示为 Tier 0，不伪装为平台维护。

**依赖关系**

依赖 F-001、F-006。

---

### F-010 平台原生 Agent 发布

**用户问题**

开发者需要把自己的 Agent 发布到平台，并进入后续审计、展示、租赁流程。

**功能范围**

提供平台内发布流程：

- 填写 Agent 名称、简介、分类、标签。
- 提交 manifest URL。
- 选择是否申请审计。
- 选择是否上架租赁。
- 提交质押或注册交易。

第一版可以将“发布”和“申请审计”绑定到现有 `stake` 流程。

**当前基础**

- `AgentAuditRegistryV3.stake(agentName, manifestUrl)` 已能注册并触发 Pending 审计。
- `docs/agent-integration-guide.md` 已描述 manifest 格式。

**交付物**

- 发布页面。
- Manifest 校验提示。
- 提交交易入口。
- 发布成功页。
- 发布后进入 Agent 详情页。

**验收标准**

- 开发者可以通过页面提交一个 Agent。
- 提交后链上产生 `AuditRequested`。
- 新 Agent 出现在平台原生列表。
- 发布失败时显示明确原因。

**依赖关系**

依赖钱包/合约写操作接入。

---

### F-011 自助沙箱审计入口

**用户问题**

开发者希望对自己的 Agent 进行可信度评估，并把结果关联到 Agent 页面。

**功能范围**

提供审计入口和状态展示：

- 对已发布 Agent 发起审计。
- 展示 Pending、Running、Passed、Failed。
- 审计完成后展示信任报告。
- 失败时展示失败原因和重试建议。

第一版可以复用 `AuditRequested` 事件驱动，不要求额外任务队列 UI。

**当前基础**

- `processAuditRequested` 已能处理审计请求。
- `writeAuditResult` 已能写回合约。
- `AuditReportPage` 已能展示报告。

**交付物**

- 详情页或开发者页的“申请审计”入口。
- 审计状态组件。
- 审计完成后的报告跳转。
- 失败原因说明。

**验收标准**

- 用户能对平台原生 Agent 发起审计。
- 审计状态可被前端读取并展示。
- 审计完成后，详情页出现最新审计摘要。
- 报告页可打开并验证报告哈希或证据。

**依赖关系**

依赖 F-010 和现有 listener 部署。

---

### F-012 租赁/购买/使用入口

**用户问题**

用户在确认某个平台原生 Agent 值得尝试后，需要直接租赁、购买或进入使用入口。

**功能范围**

在 Agent 详情页提供：

- 租赁按钮。
- 买断按钮。
- 访问权状态。
- 访问成功后的使用入口。
- 交易历史或访问记录。

第一版只支持平台原生 Agent。

**当前基础**

- `AgentMarketplace` 已有 `rentAgent`、`buyAgent`、`hasAccess`。
- 前端已有 `PricingCard` 和 `AccessHistoryCard`。

**交付物**

- 租赁/购买交易按钮。
- 钱包连接与交易状态。
- hasAccess 状态展示。
- 使用入口 URL 或说明。

**验收标准**

- 用户可完成一次租赁交易。
- 交易后 `hasAccess` 返回 true。
- 详情页显示访问权状态。
- 无价格配置时，不展示不可点击的购买按钮。

**依赖关系**

依赖钱包/合约写操作接入。

---

### F-013 评论与六维评分闭环

**用户问题**

用户需要看到真实使用反馈；有访问权的用户也应能提交评价。

**功能范围**

提供评论展示和提交：

- 查看评论。
- 提交六维评分。
- 提交文字评论。
- 评论与 Agent 绑定。
- 仅有访问权用户可提交。

**当前基础**

- `AgentReviewRegistry` 已有访问权门禁和六维评分。
- `reviewCommentApi` 可存储链下评论全文。
- 前端有 `ReviewSection`、`ReviewSubmitForm`、`useAgentReviews`。

**交付物**

- 完整接入 review client、marketplace client、user address。
- 评论提交表单。
- 评论哈希与链下文本存储关联。
- 评论提交成功后的刷新。

**验收标准**

- 无访问权用户不能评论，并看到原因。
- 有访问权用户可以提交一次评论。
- 重复评论被阻止。
- 评论提交后，评分分布和评论列表更新。

**依赖关系**

依赖 F-012。

---

### F-014 平台信任记录聚合

**用户问题**

单独的信誉分不足以表达“平台内发生了什么可信事件”，用户需要看到可解释的信任记录。

**功能范围**

建立统一信任记录：

- 审计完成。
- 审计通过或失败。
- attestation 可用。
- 用户租赁/购买。
- 用户评论。
- 平台观察。
- 高等级验证。

第一版可以在前端聚合展示，不要求修改合约信誉算法。

**当前基础**

- `ReputationRecord` 有审计和申诉相关信誉。
- `AgentMarketplace` 有访问记录。
- `AgentReviewRegistry` 有评价记录。
- 外部观察待 F-009 提供。

**交付物**

- TrustRecord 数据结构。
- 详情页时间线。
- Agent 卡片上的信任摘要。

**验收标准**

- 用户能看到某个 Agent 的关键可信事件。
- 审计、租赁、评论、观察来源清晰区分。
- 不将信誉分表述为“全世界真实信誉”，只表述为“平台信任记录”。

**依赖关系**

依赖 F-009、F-012、F-013。

---

### F-015 信任报告页产品化

**用户问题**

当前审计报告偏技术证明，普通用户难以理解其对选择 Agent 的意义。

**功能范围**

将审计报告页分成两层：

- 用户摘要：结论、分数、风险、建议。
- 技术证据：report hash、evidence root、attestation hash、资源指标、网络行为。

**当前基础**

- `AuditReportPage` 已有报告展示。
- `buildAuditReport`、`evidence`、`attestation` 已有底层能力。

**交付物**

- 报告摘要模块。
- 风险解释模块。
- 技术证据折叠模块。
- “这份报告意味着什么”说明。

**验收标准**

- 非技术用户能在 30 秒内理解审计结论。
- 技术用户仍能看到完整证据字段。
- 报告页能回链到 Agent 详情页和信任层级。

**依赖关系**

依赖 F-004、F-006。

---

## 8. P2 功能包

### F-016 收藏与候选清单

**用户问题**

用户在选型过程中可能暂时不决策，需要保存候选 Agent 后续比较。

**功能范围**

- 收藏 Agent。
- 查看候选清单。
- 从候选清单进入 Compare。
- 本地存储即可，不要求账号系统。

**验收标准**

- 用户刷新页面后收藏仍保留。
- 收藏清单可以移除 Agent。
- 清单中的 Agent 可以一键进入 Compare。

**依赖关系**

依赖 F-007。

---

### F-017 运营维护后台或数据工作流

**用户问题**

平台运营方需要持续维护重点 Agent 的内容，否则无法形成区别于大模型推荐的壁垒。

**功能范围**

第一阶段可不做后台管理 UI，但必须有可维护工作流：

- 重点 Agent 数据文件。
- 外部观察记录文件。
- 安装导航文件。
- 内容校验脚本。
- 变更 review 规则。

**验收标准**

- 运营方可以新增一个重点维护 Agent，不需要改多个页面组件。
- 数据缺字段时，构建或测试能提示。
- 每个重点维护 Agent 至少有基础资料、风险、场景、安装导航、最近观察。

**依赖关系**

依赖 F-001、F-005、F-009。

---

### F-018 指标与转化埋点

**用户问题**

平台需要知道哪些功能真正帮助用户完成决策，以便持续优化。

**功能范围**

追踪以下事件：

- 搜索。
- 筛选。
- 进入详情页。
- 加入 Compare。
- 查看安装导航。
- 跳转官方链接。
- 发起租赁。
- 提交评论。

**验收标准**

- 每个核心路径有事件记录。
- 可以统计搜索到详情页转化率。
- 可以统计 Compare 使用率。
- 可以统计安装导航点击率。

**依赖关系**

依赖核心页面稳定后接入。

## 9. 建议实施顺序

### 阶段一：把产品从“链上市场”改造成“选型平台”

优先完成：

1. F-001 Agent 内容数据模型。
2. F-006 信任层级映射。
3. F-002 首页定位与入口重构。
4. F-003 搜索与筛选体验。
5. F-004 Agent 决策型详情页。
6. F-005 安装与接入导航。

阶段验收：用户可以搜索或按场景浏览 Agent，并在详情页完成“是否值得尝试”的判断。

### 阶段二：补齐决策闭环

优先完成：

1. F-007 Agent 横向 Compare。
2. F-008 需求输入到候选推荐。
3. F-009 持续更新外部视图。
4. F-015 信任报告页产品化。

阶段验收：用户可以从“描述需求”走到“推荐候选、对比、查看报告、开始使用”。

### 阶段三：产品化平台原生生态

优先完成：

1. F-010 平台原生 Agent 发布。
2. F-011 自助沙箱审计入口。
3. F-012 租赁/购买/使用入口。
4. F-013 评论与六维评分闭环。
5. F-014 平台信任记录聚合。

阶段验收：开发者可以发布 Agent，用户可以审计、租赁、评论并查看完整信任记录。

### 阶段四：运营与增长

优先完成：

1. F-016 收藏与候选清单。
2. F-017 运营维护后台或数据工作流。
3. F-018 指标与转化埋点。

阶段验收：平台具备持续维护内容、观察用户行为并优化转化的能力。

## 10. MVP 验收清单

MVP 不要求一次性完成全部 18 个功能包，但必须完成以下闭环：

- 用户可以看到 curated、listed、native 三类 Agent。
- 用户可以通过搜索、场景、风险、复杂度筛选候选。
- 用户可以进入详情页看到适用场景、不适用场景、风险、安装导航。
- 用户可以理解 Agent 的信任层级及其原因。
- 用户可以对 2-4 个 Agent 做横向 Compare。
- 用户可以从详情页跳转官方入口或查看平台原生 Agent 的租赁/审计能力。
- 真实租赁/购买交易不属于 MVP 必须项；若未完成 F-012，MVP 只要求展示价格、访问权说明和后续入口。
- 重点维护 Agent 至少覆盖 10-20 个高质量样例。
- 每个重点维护 Agent 都有平台整理的起步建议，而不是只有官方链接。

---

## 11. 暂不纳入 MVP 的内容

以下内容不建议放入第一版闭环：

- 完整 CMS 后台。
- 大规模自动爬取和自动更新。
- 复杂 AI 推荐模型。
- 企业团队协作。
- 企业对比导出。
- 更复杂的链上信誉算法重构。
- 全量外部 Agent 的审计接入。

这些能力可以在核心选型路径跑通后再规划。

---

## 12. 一句话重整结论

AgentLens 当前最有价值的技术资产是“可信审计与链上记录底座”；下一阶段最应该补齐的是“用户选型决策前台”。因此，需求应先围绕搜索、详情、安装导航、信任层级和 Compare 建立可用闭环，再逐步把现有发布、审计、评论、租赁能力产品化为平台原生 Agent 生态。
